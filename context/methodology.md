# 方法论

以下整理「常用方法论」：按与 `expectations.md` 相同的口径组织，每条含 **解释**、**场景**、**不适用场景**、**使用步骤** 与 **常见误区**。标题样式、分隔方式与 `expectations.md` 保持一致，并在标题中标注其主要归属维度。

---

## 1. 5 why（产品）

**解释**  
`5 why` 是一种通过连续追问“为什么”来逐层逼近问题根因的方法，用来避免只停留在表面现象。

**场景**  
适合用于分析重复发生的问题、定位体验断点背后的真实原因、复盘项目失败原因，以及拆解看似简单但实际成因复杂的现象。

**不适用场景**  
不适合用于强依赖外部环境、存在多个并行原因的大型系统问题，也不适合在缺少基本事实和证据时凭感觉硬追问。

**使用步骤**  
1. 先明确当前看到的表面问题，不要一开始就跳到解决方案。  
2. 围绕这个问题连续追问“为什么”，每一层回答都尽量基于事实。  
3. 追问到无法继续拆解，或已经触达可行动的根因时停止。  
4. 检查这个根因是否真的能解释前面的现象，避免因果断裂。  
5. 基于根因再去讨论应对方案，而不是针对表面症状修补。

**常见误区**  
- 把“问满五次”当成目标，实际上问到根因即可，不一定必须五次。  
- 每一层都靠主观猜测，没有用数据、访谈或事实校验。  
- 只找单一原因，忽略复杂问题常常由多个因素共同造成。

## 2. 第一性原理（产品）

**解释**  
第一性原理是一种把问题拆到最基础事实与不可再分的前提，再从这些基础重新构建结论和方案的思考方式。

**场景**  
适合用于产品定位、方案重构、打破惯性路径、识别伪需求，以及在复杂决策中区分“必须成立的本质”与“习惯性的假设”。

**不适用场景**  
不适合用于时间极紧、只需局部优化的小问题，也不适合在基础事实尚不清楚时空谈“本质”，容易沦为抽象口号。

**使用步骤**  
1. 先明确你正在解决的核心问题，避免问题本身就定义错了。  
2. 写出这个问题背后默认成立的常见假设和行业惯例。  
3. 逐条判断哪些是假设，哪些是真正不可回避的基础事实。  
4. 从基础事实重新推导可能的解法，而不是沿用已有方案。  
5. 对新推导出的方案做可行性验证，避免停留在理论层面。

**常见误区**  
- 把“推翻常识”误认为第一性原理，实际上它强调的是回到基础事实。  
- 只做抽象思辨，不落到具体产品决策和验证上。  
- 把个人偏好当成“本质”，忽略用户真实需求和现实约束。

## 3. HEART（产品）

**解释**  
`HEART` 是 Google 提出的体验衡量框架，用幸福感、参与度、采纳度、留存率和任务成功率五个维度来评估产品体验。

**场景**  
适合用于建立体验指标体系、评估体验改版是否有效、把抽象的“体验变好”转化为可跟踪的衡量方式，以及帮助团队统一关注的体验目标。

**不适用场景**  
不适合直接替代业务指标分析，也不适合在没有清晰用户行为数据、没有明确目标动作定义时机械套用。

**使用步骤**  
1. 先明确你要衡量的是哪一段用户体验，而不是笼统衡量整个产品。  
2. 在 `HEART` 五个维度里挑出最相关的体验目标。  
3. 把体验目标翻译成可观察、可计算的具体信号或指标。  
4. 建立改版前后的对照方式，观察指标是否真实变化。  
5. 结合业务背景解释结果，避免只看数字不看语境。

**常见误区**  
- 把五个维度全部强行使用，实际上应只选择当前最相关的部分。  
- 只定义宏观指标，没有把指标落到具体用户行为。  
- 把体验指标和商业指标混在一起，导致判断口径不清。

## 4. 用户旅程画布（用户画像）

**解释**  
用户旅程画布是一种按用户任务过程梳理阶段、行为、触点、情绪、痛点与机会点的方法，用来理解完整体验链路。

**场景**  
适合用于分析用户在完整流程中的体验变化、识别关键流失点、发现跨触点问题，以及在产品、设计、运营之间建立统一的用户视角。

**不适用场景**  
不适合在用户路径极其简单、问题只发生在单个页面细节时优先使用，也不适合完全没有用户研究材料时凭空绘制。

**使用步骤**  
1. 明确要分析的是哪类用户、在哪个具体任务场景下的旅程。  
2. 按时间顺序拆出关键阶段和关键触点。  
3. 记录每个阶段用户在做什么、想什么、感受如何。  
4. 标出痛点、卡点、情绪波动和流失风险。  
5. 从关键节点提炼出可优化的机会点和优先级。

**常见误区**  
- 只画流程，不写用户情绪、动机和阻碍。  
- 把团队内部想象当成用户真实旅程，没有研究依据。  
- 画得过大过全，最后无法落到可执行优化点。

## 5. 尼尔森（体验设计）

**解释**  
尼尔森原则通常指 Jakob Nielsen 提出的 10 条可用性启发式原则，用来快速检查界面和交互是否符合基本可用性规律。

**场景**  
适合用于设计评审、交互走查、可用性问题排查，以及在没有大规模用户测试时先做一轮专家视角的基础检查。

**不适用场景**  
不适合代替真实用户研究，也不适合用于判断一个产品是否具备市场价值或商业可行性。

**使用步骤**  
1. 明确要检查的页面、流程或任务场景。  
2. 按尼尔森 10 条原则逐项走查界面与交互。  
3. 记录不符合原则的地方，并描述具体问题表现。  
4. 根据影响范围和严重程度给问题排序。  
5. 将问题转化为可以执行的设计优化建议。

**常见误区**  
- 只背原则名称，不结合具体任务场景判断。  
- 把原则当成绝对规范，忽略产品类型和上下文差异。  
- 找到了问题却没有进一步评估严重程度和优先级。

## 6. 格式塔原理（体验设计）

**解释**  
格式塔原理是一组关于人如何感知和组织视觉信息的规律，常用于指导界面中的分组、层级、对齐、连续性和整体感设计。

**场景**  
适合用于界面布局设计、信息层级梳理、组件分组、视觉引导和提升页面可读性，尤其适合解决“信息很多但看起来很乱”的问题。

**不适用场景**  
不适合用来代替交互逻辑设计、功能策略判断或用户需求分析，它更多解决的是视觉组织问题，而不是产品方向问题。

**使用步骤**  
1. 先明确页面中哪些信息应该被用户优先感知。  
2. 检查当前布局在接近性、相似性、连续性、闭合性等方面是否清晰。  
3. 调整模块间距、样式一致性、对齐关系和视觉节奏。  
4. 验证用户是否能快速识别分组、重点和操作路径。  
5. 根据场景继续微调，兼顾美观与理解效率。

**常见误区**  
- 把格式塔原理理解成纯视觉审美，而忽略它服务于信息理解。  
- 只关注单个模块是否好看，没有考虑整体页面的组织关系。  
- 机械套用某条原则，导致界面失去灵活性和真实场景适配。
