# 值得深挖 — 编辑口径与执行规则

本文档供 `pipeline/prompts/processing.py`、`generation.py` 与人工审核对齐；简报中 **## 值得深挖** 节仅展示系统短名单（0～2 条），**禁止凑数**。

---

## 「新」的裁决（硬门槛 + 排序）

1. **硬门槛**：须为 **官方或可信渠道发布**（changelog、官方博客、应用内更新说明、正式新闻稿等）。**不满足者一律不进入** 0～2 条深挖短名单（仅二手转载、传闻、无法核验者，即使频次高也不入选）。
2. **在已过硬门槛的候选中**，按下列顺序 **tie-break**（不替代硬门槛）：
   - **当日原始抓取**中，同一 `anchor_product`（及必要时 capability）相关条目 **出现次数更多** 者优先；
   - 仍并列时，比较 **近 N 天**（见 `config.settings.DEEP_DIVE_SIGNAL_DAYS`，默认 7）内在 `data/raw/` 中合并统计的 **出现次数** 更高者。
3. **无任何合格候选时，深挖条数为 0**：省略 `## 值得深挖` 整节（不写「暂无」占位亦可，与生成 prompt 一致即可）。

系统会在每条 processing 记录上写入 `signal_same_day_count`、`signal_ndays_count` 供排序；**是否官宣**由模型输出 `release_evidence`，并由代码与 `deep_dive_fit` 联合过滤。

---

## 数量与合并

- **深挖短名单：0～2 条**。宁可不输出，不可用弱相关条目凑满。
- **同一宿主产品多条不同能力**：合并为 **一条** 展示（`anchor_capability` 用顿号或「、」连接多个能力名），占 1 个名额。

---

## 准入：什么叫「工具」（Tool）

工具指：以 **可反复使用的产品形态** 交付、主要用途是帮用户完成某类 **具体任务** 的软件或服务（含插件、独立站、CLI、IDE 扩展、Figma 插件等），通常具备相对清晰的 **入口（官网/应用）+ 能力边界 + 使用场景**。

**满足「工具」的常见特征**（满足多数即可）：

- 有可指认的产品名（如 Claude、Figma、Cursor、Gemini、OpenClaw、Lovable、Lovart、Liblib 等），不是泛指「AI 能帮设计」这类能力描述。可参考编辑维护的 **AI 工具分类清单**（聊天 / 写作 / 编程 / 设计 / 视频等）。
- 用户能回答：「我用它来干什么？」（例如：生成界面、改稿、出图、写 spec、做调研、管组件库等）。
- 相对可试用或可订阅（内测/候补也可，但应有 **可验证的产品载体**，而非仅概念）。

**不算「工具」的典型情况**：

- 只有方法论、工作流分享，没有可安装/可打开的具体产品。
- 某大厂「整体战略/愿景」通稿，未落到可命名的独立能力或模块。
- 开源纯论文/模型权重且无面向设计/产品工作流的明确产品壳（默认深挖列表以 **产品向** 为主）。

---

## 准入：什么叫「产品新功能」（New product capability）

产品新功能指：在 **已有、已命名的宿主产品** 上，新增的、可单独说明的 **能力单元**（模块、模式、入口、API、工作流），对用户行为或产出有可感知变化，且通常有 **官方或可信渠道的发布说明**。

**识别时锚定**：

1. **宿主产品**（如 Claude、Figma、Notion）。
2. **能力名称或用户可见入口**（如 Design 模式、产品升级说明中的命名——**以官方为准**）。

**不算「产品新功能」**：仅文案/品牌/定价调整；不构成新能力的小修小补；路线图/预告/泄露/传闻且无可用形态。

---

## 「AI 强相关」— 正面定义

满足下列 **至少一条**，且与 **工具或产品新功能** 绑定，则 `ai_involvement` 可为 `core`：

1. 核心能力依赖生成式/多模态模型等，且为产品 **主路径或重要路径**。
2. 明确面向 **设计/创意/产品工作流**（界面、原型、设计系统、用研、文案、交付等）。
3. **人机协作形态清晰**（对话改稿、Agent 多步、自然语言到可交付物等），且与 **具体产品功能** 绑定。

「强相关」看的是 **耦合度 + 与读者工作流的关联度**，不是标题里出现 “AI” 二字。

---

## 明确排除（不进 0～2 深挖短名单）

下列类型 **原则上排除**（素材再好也最多进正文其它节）：

- 纯融资/估值/收购/人事，且 **无** 同日可验证的产品或功能发布。
- 纯观点/评论/行业综述/预测，无具体工具或新功能可分析。
- 非产品更新：政策、诉讼、伦理争议、排行榜盘点、纯数据报告等。
- 泛平台动态但未落到 **设计/产品工作流工具** 上的具体能力或集成。
- 仅活动/课程/播客/采访，无独家可核查的产品/功能披露。
- 重复炒作、无实质增量。
- **无法指认** 具体产品或具体新能力。

---

## `deep_dive_fit` 与灰区

- **strong**：明确符合工具/产品新功能 + `ai_involvement=core` + 未命中排除 + **官宣证据充分** → 可进系统短名单（仍受 0～2 与排序约束）。
- **weak**：值得后续跟进但 **证据不足、或增量偏工具性小改版、或官宣边界模糊**（例：某编辑器多文件编辑类更新）→ **不进入** 深挖短名单；可在正文「工具与产品能力」等节正常呈现。
- **none**：不符合深挖定位。

---

## 与 processing 字段的对应

| 编辑概念 | 字段 |
|----------|------|
| 工具 vs 宿主新能力 vs 非产品 | `release_kind`: `standalone_tool` / `host_feature` / `not_product` |
| 宿主/主产品规范名 | `anchor_product` |
| 能力短标签（宿主新能力时建议非空） | `anchor_capability` |
| AI 关联强度 | `ai_involvement`: `core` / `adjacent` / `none` |
| 是否适合进深挖池 | `deep_dive_fit`: `strong` / `weak` / `none` |
| 官宣证据类型 | `release_evidence`（见 processing 系统提示中的枚举） |
| 当日/近 N 天频次（代码写入） | `signal_same_day_count`, `signal_ndays_count` |

**深挖短名单 100% 来自** 当日 `processing.json` 中已合并、已 enrich 的条目；简报展示名与口播选题统一为 **`standard_display_name`**（由代码生成：`anchor_product` + ` · ` + `anchor_capability`，合并多能力时为 `能力1、能力2`）。

---

## 用户触发长文（约定）

- 生成简报：自然语言如「帮我生成今天的简报」对应跑 `pipeline.main` / `/daily-briefing`。
- 针对深挖写长文：优先使用简报中的 **标准展示名**（如 `Claude · Design`）；可维护 **别名表** 映射口语到标准名（实现见 `tools/generate_article` 后续扩展）。
